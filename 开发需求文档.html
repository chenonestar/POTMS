1. 项目概述

1.1 项目背景与规模

根据国家对公职人员因私出国（境）的管理规定，需对纳入备案管理范围的人员实行全流程闭环管理。本系统用于支撑从人员备案、证照登记、出国（境）申请审批到信息变更的全部线上业务，确保合规、高效、可追溯。

规模指标说明
备案人员规模约 500 人
系统用户数1 人统管（单用户，管理员全权操作）
审批模式无多流程审批，由管理员直接录入、登记、归档
并发访问单用户操作，无并发压力

📌 系统定位为小型单机管理系统，无需复杂的工作流引擎或多角色权限体系，以简洁实用为设计原则。

1.2 核心业务流程总览

人员备案 → 证照登记 → 出国申请？ → 领用/申办证件 → 填明细表&上传材料 → 审批归档

信息变更流程：撤控备案 → 重新登记备案（形成子环）

1.3 系统范围与目标

  • 实现备案人员信息的录入、查询、修改、导出。
  • 实现三类出入境证照的集中登记与状态跟踪（护照 / 港澳通行证 / 台湾通行证）。
  • 实现因私出国（境）申请的全流程线上审批，包括证件领用、材料上传与明细填报。
  • 实现备案信息变更的"撤控—重报"双步操作。
  • 所有表单支持 Excel 导出 与 在线打印。

2. 用户角色

单用户管理模式

本系统规模较小（约500名备案人员），采用单用户统管模式，不设多角色多级审批。

角色职责描述核心权限
管理员（唯一用户） 统筹所有备案管理、证照登记、出国申请、信息变更、数据导出打印等全部业务操作 全部功能权限（录入、查询、修改、撤控、导出、打印、数据维护）

📌 系统仅设置单一管理员账户，登录后直接进入功能界面，无需角色切换、权限分配或审批流转。

📌 因系统为单用户统管，各业务表单无需手动填写"操作人"，系统在保存时自动记录当前登录管理员为操作人并写入操作日志。

5. 业务模块 — 因私出国（境）申请

5.1 申请流程分支

备案人员发起申请 → 是否已有有效证件？

✅ 路径A：已有证件

  1. 办理证件领用手续（系统中记录领用）
  2. 填写《因私出国（境）人员明细表》
  3. 上传扫描件：
    • 《因私出国（境）个人申请报告》
    • 《因私出国（境）审批表》

📌 路径B：无有效证件

  1. 线下办理《关于同意申办出入境证件的函》
  2. 凭函向公安机关出入境管理部门申办证件
  3. 证件办理完成后，填写《因私出国（境）人员明细表》
  4. 上传扫描件（含"同意申办函"，共3项）

5.3 处理流程（单用户模式）

管理员统一处理出国（境）申请的全流程——根据线下已签署的纸质材料，在系统中完成信息录入与归档：

  1. 录入：管理员根据《个人申请报告》《审批表》等线下已签批材料，填写《因私出国（境）人员明细表》并上传扫描附件。
  2. 证照操作：已有证件→记录领用日期；无证件→记录"同意申办函"信息及后续证件号码。
  3. 归档：信息填写完毕后直接保存归档，系统记录操作时间及操作人。

📌 所有审批环节在线下完成，系统仅作为信息登记与电子归档平台，不做线上审批流转。

7. 通用功能需求

7.1 Excel 导出 REQ-004

  • 所有表单（5类表）均支持一键导出为 Excel（.xlsx）格式。
  • 导出支持：按筛选条件导出、全量导出、选中行导出。
  • 导出文件须包含：表头、数据行、填表说明（作为 Sheet2 或页脚行）。
  • 导出时自动生成文件名：表单名称_导出日期_操作人.xlsx。

7.2 在线打印 REQ-005

  • 所有表单支持浏览器端在线打印（调用浏览器打印功能）。
  • 打印样式须适配 A4 纸张，自动去除导航栏、侧边栏等无关元素。
  • 支持打印预览，可调整页边距、缩放比例。
  • 打印内容须包含：表单标题、数据内容、填表人/审核人签章栏、打印日期。

7.3 查询与筛选 REQ-006

  • 全局搜索：按姓名、身份证号、单位进行全系统快速检索。
  • 人员选择器：在证照登记、出国申请等需选择备案人员处，提供按姓名 / 身份证号 / 单位实时过滤的搜索框，适应约 500 人规模下的快速定位。
  • 高级筛选：按政治面貌、职级、户口所在地、备案状态、证照类型、出行日期范围等多条件组合筛选。
  • 列表页支持：排序、分页、自定义列显示/隐藏。

7.4 数据导入 REQ-007

  • 支持批量导入备案人员信息（Excel模板导入）。
  • 导入时进行逐行校验（身份证校验、必填项检查、枚举值匹配），错误行生成错误报告，正确行正常入库。
  • 提供导入模板下载功能。

7.5 操作日志与审计 REQ-008

  • 记录所有关键操作：备案录入/修改/撤控、证照登记/领用/归还、申请提交/审批/退回、数据导出。
  • 日志内容：操作人、操作时间、IP地址、操作类型、操作对象（人员ID）、变更前后数据快照。
  • 日志不可删除、不可篡改，支持按时间范围和操作类型查询。

7.6 提醒功能 REQ-009

  • 证照有效期到期前30天，在系统首页以醒目提示条展示即将到期的证照信息。
  • 证件领用后超过计划归还日期未归还，系统首页高亮标记逾期未还记录。
  • 以上提醒均在管理员登录后的首页仪表盘集中展示，无需外部消息推送。

7.7 首页仪表盘 REQ-010

  • 备案人员总数及分类统计（按单位、政治面貌、职级等维度）。
  • 证照状态总览卡片（在库 / 领用中 / 逾期未还，点击可跳转明细列表）。
  • 近期出国（境）计划列表（按出行日期排序）。
  • 证照到期 & 归还逾期预警列表（置顶显示）。

9. 非功能需求

9.1 安全性

  • 身份认证：管理员用户名/密码登录，密码哈希存储（bcrypt）。
  • 会话管理：基于 Session 或 JWT Token，支持登录超时自动退出。
  • 数据脱敏：身份证号在前端列表展示时部分掩码（如 330***********1234）。
  • 附件安全：上传文件限制为 PDF 格式，单文件大小 ≤10MB。
  • 备份安全：SQLite 数据库文件定期备份至指定目录。

9.2 性能

  • 列表页查询响应时间：≤1秒（500条记录级别，SQLite 单机无压力）。
  • Excel 导出：500条全量导出应在 3 秒内完成。
  • 附件上传：支持 PDF 文件上传，单文件 ≤10MB。
  • 并发：单用户操作，无需考虑并发控制。

9.3 可用性

  • SQLite 数据库文件每日自动备份至 backup/ 子目录，保留最近 30 天。
  • 关键操作提供二次确认弹窗（如撤控、删除记录）。
  • 表单页支持暂存草稿功能，防止填写中断丢失数据。

9.4 兼容性

  • 浏览器：Chrome 90+、Edge 90+、Firefox 90+、国产浏览器（360安全浏览器、QQ浏览器）最新版。
  • 分辨率：最低支持 1366×768，推荐 1920×1080。
  • 操作系统：Windows 10/11（64位）。
  • 运行模式：本地启动 Web 服务，浏览器访问 http://localhost:端口。

Web 框架 Flask / FastAPI + Jinja2 模板引擎，生态极其成熟，前后端一体开发效率极高 net/http + html/template 标准库即用，或 Gin/Echo 框架，模板语法较基础
SQLite 支持 ✅ 内置 sqlite3 标准库，无需 CGO 编译，纯 Python 驱动（或 aiosqlite）即可 ⚠️ 主流驱动 mattn/go-sqlite3 依赖 CGO 编译；纯 Go 方案（modernc.org/sqlite）可用但生态较新
Excel 生成 ✅ openpyxl：功能最全，支持样式、合并单元格、打印设置、多 Sheet，完全满足5类表单导出需求 ⚠️ excelize：Go 生态最好的 Excel 库，基本功能齐全，但复杂样式和打印设置能力弱于 openpyxl
PDF 处理 ✅ reportlab / WeasyPrint / pdfkit，多种成熟方案可选 ⚠️ gofpdf / pdfcpu，功能较基础
表单校验 ✅ WTForms / Pydantic，声明式校验，身份证校验等规则库丰富 ⚠️ go-playground/validator，够用但需手写较多自定义规则
开发效率 ✅ 语法简洁，热重载开发，从需求到原型的周期极短 ⚠️ 语法较冗长，错误处理显式（if err != nil），开发速度慢于 Python
部署便捷性 ⚠️ 需 Python 解释器 + pip 依赖；可用 PyInstaller 打包为单个 .exe ✅ 编译为单文件 .exe，无任何运行时依赖，拷贝即运行
运行性能 ⚠️ 解释型语言，但 500 人数据量完全无感知（毫秒级查询） ✅ 编译型语言，高性能，但在此规模下优势无法体现
打包体积 ⚠️ PyInstaller 打包后约 30–60 MB ✅ 编译后约 8–15 MB
学习成本 ✅ 语法简单，上手快，生态资料丰富 ⚠️ 接口设计、错误处理、指针等概念对新手有门槛

🏆 推荐结论：选用 Python

核心理由：
  1. Excel 导出是硬需求 — 5 类表单需要精确的样式、合并单元格、打印排版，openpyxl 是业内最成熟的方案，Go 的 excelize 在复杂格式方面差距明显。
  2. SQLite 无编译痛点 — Python 标准库内置 sqlite3，真正实现"零依赖"，而 Go 的纯 Go SQLite 驱动仍不够成熟。
  3. 开发效率优先 — 单用户、500 人、无并发的小型系统，开发速度和维护简便性远比运行性能重要。
  4. 部署可通过 PyInstaller 弥补 — 开发完成后打包为单个 .exe，用户体验接近 Go 的编译分发。

推荐技术组合：Python 3.12+ + Flask + SQLite3 + openpyxl + Jinja2 + Bootstrap 5（前端 UI）
打包分发：PyInstaller 生成单文件 .exe，内置 Python 运行时 + SQLite 数据库模板
