一、审定范围与结论

本次审定基于 main 分支最新代码（含 PR #1「系统评审整改 + 质量与安全加固」与 PR #2「运行反馈整改 + 视口自适应分页」全部内容），逐文件核查了数据库层、蓝图路由、模板、前端脚本、配置与测试，每一条发现均经代码实证，非推测。

结论：系统当前无阻断性缺陷、无已知功能性 bug，pytest 44 用例全绿。本报告聚焦"还能更好"的部分：识别出 16 项可优化点，其中仅 1 项建议优先处理（登录防爆破），其余为中低优先级的加固、加分项。全部优化均可在不引入新依赖、不增加 exe 打包体积的前提下完成。

二、已达良好水准的方面（无需再动）

以下方面经核查已是良好实践，本轮无发现、无需改动，列出以明确审定覆盖面：

  • 安全基线：CSRF 全覆盖（常量时间比对）、bcrypt 密码哈希（旧哈希透明升级）、SQL 全参数化 + 表名白名单、上传文件 UUID 重命名、send_from_directory 防路径穿越、身份证列表脱敏。
  • 数据层：WAL 日志模式、外键约束开启、幂等迁移（升级免手动改库）、每日自动备份保留 30 天。
  • 时间处理：统一存 UTC / 显示转本地（可配置偏移），符合教科书标准。
  • 查询效率：核查 get_personnel_options 等热点函数，无 N+1 查询（人员+证件两条查询建映射，写法正确）。
  • 工程化：pytest 44 用例（校验/工作日/逾期/迁移/分页/时区）、公共校验函数（DRY）、TypedDict 与路由返回注解、测试与 exe 打包物理隔离。
  • 体验：视口自适应分页（跨屏幕/缩放）、树状部门选择、草稿暂存、列显示自定义、附件预览、悬停查看全文。

三、安全类优化项

编号问题实证与建议优先级工作量
S1登录无防爆破机制 实证：auth.py 的 login 路由对失败尝试无任何计数、锁定或延时，可无限次试错口令。内网单机风险虽低，但这是当前最大的安全敞口（尤其若管理员未改默认口令）。
建议：内存级失败计数——同一来源连续失败 5 次锁定 10 分钟（或按失败次数递增延时 1s/2s/4s…），并写入操作日志。纯标准库实现。		 P1约半天
S2附件上传仅校验扩展名 实证：travel.py 上传仅比对 ALLOWED_EXTENSIONS={"pdf"}，改名的任意文件可入库（预览时按 application/pdf 输出，实际危害有限，属纵深防御缺口）。
建议：保存前读文件头校验 %PDF- 魔数，不符即拒绝。约 5 行代码。		 P2约 1 小时
S3Session Cookie 安全标志未显式声明 实证：config.py 未设置 SESSION_COOKIE_HTTPONLY / SAMESITE（Flask 默认 HttpOnly=True，SameSite 依赖浏览器默认）。
建议：显式加 SESSION_COOKIE_SAMESITE="Lax"、SESSION_COOKIE_HTTPONLY=True，不依赖默认行为。两行配置。		 P310 分钟
S4备份/导出文件明文含敏感数据 实证：backup/ 与 exports/ 下的 db/xlsx 含全量身份证号，同机任何账户可读。属部署环境管理问题而非代码缺陷。
建议：在 README 部署章节增加提示——运行目录设 NTFS 权限/启用 BitLocker，导出文件用毕删除。		 P3文档
S5草稿 localStorage 含敏感字段 实证：表单草稿（含身份证号）存浏览器 localStorage，登出后仍保留；共用电脑场景有暴露面。
建议：退出登录时清空 draft_* 键。几行 JS。		 P3约 1 小时

四、性能类优化项

编号问题实证与建议优先级工作量
F1数据库零索引 实证：database.py 无任何 CREATE INDEX。身份证查重、status='active' 过滤、attachments.travel_id 关联、日志按时间筛选均为全表扫描。
影响评估：500 人规模下毫秒级，当前无感；属"面向未来"的加固。
建议：迁移中一次性补 5~6 个索引（personnel_filing.id_number/status、attachments.travel_id、operation_logs.created_at、travel_details.personnel_filing_id 等），幂等 CREATE INDEX IF NOT EXISTS。		 P2约 2 小时
F2表单模板重复调用查询型全局函数 实证：info_form.html 中 org_flat() 被调用 3 次、dict_opts() 多次，每次一条 SQL，单次渲染约 10 条小查询。本地 SQLite 开销极小。
建议：模板顶部 {% set orgs = org_flat() %} 缓存为一次调用。		 P3约 1 小时
F3首页每次访问都执行备份检查 实证：dashboard.index 每次请求调用 run_daily_backup()（幂等跳过，但每次做文件存在检查与目录扫描）。
建议：进程内记录"今日已检查"标记，当天第二次起直接跳过。		 P3约 30 分钟

五、健壮性与运维类优化项

编号问题实证与建议优先级工作量
R1无自定义 404 / 500 错误页 实证：仅注册了 CSRF 的 400 处理器；404/500 走 Flask 默认英文页面，与系统风格不符，出错时对操作员不友好。
建议：增加中文 404/500 模板（含"返回首页"），500 页同时写入日志。		 P2约半天
R2exports/ 导出文件只增不删 实证：每次导出在 exports/ 落一个 xlsx，无任何清理机制，长期运行持续累积（也放大 S4 的敏感数据暴露面）。
建议：下载响应完成后即删临时文件，或沿用备份的做法保留 7 天自动清理。		 P2约 2 小时
R3备份"只备不复"——缺恢复指引 实证：README 无任何恢复（restore）说明，系统内也无恢复入口。真出事时操作员不知如何用备份还原。
建议：至少在 README 写明恢复三步（停服 → 用 backup/ 中的文件替换 data.db → 重启）；进一步可在系统内做"从备份恢复"页（含二次确认）。		 P2文档 30 分钟 / 功能约 1 天
R4操作日志无限增长、无归档手段 实证：operation_logs 只增不删（审计要求不可删，设计正确），但无导出归档功能，数年后表持续膨胀。
建议：提供"按年份导出日志归档（xlsx/csv）"功能即可，不建议自动删除。		 P3约半天

六、代码整洁类优化项

编号问题实证与建议优先级工作量
C16 处路由残留无用的 page 变量 实证：切换前端窗口化分页后，certificate/decontrol/personnel/travel(×2)/logs 的列表路由仍保留 page = request.args.get("page", …)，其中业务列表 5 处已完全无用（logs 仍在用）。属死代码，误导后来维护者。
建议：删除无用 5 处。		 P3约 20 分钟
C2函数体内 import 实证：logs.py 路由内 from config import Config、多处路由内 from datetime import datetime，可上移至模块顶部统一。
建议：顺手清理，无行为影响。		 P3约 20 分钟
